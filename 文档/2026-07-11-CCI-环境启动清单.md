# CCI 上机启动清单：SimpleTIR

日期：2026-07-11。目标是在 CCI 的 4×H100 容器中建立一个可跨重启恢复的
SimpleTIR 开发与训练环境。CCI 的 `/home` 会在约 12 小时后重置；唯一可信的
持久目录是 `/data/L202500291`。

## 不可违反的规则

- 所有会变化、需要保留或下载成本高的内容都放在 `/data/L202500291`。
- `/home` 只允许存放一次性 bootstrap 配置或指向 `/data` 的链接；它不能是唯一副本。
- 不要在确认 Docker 可用前执行项目 Dockerfile 的构建或下载大镜像。
- 不要把任何 API key、token 或登录输出发到聊天中。

建议目录：

```text
/data/L202500291/
├── SimpleTIR/              # Git 工作副本
├── codex/                  # Codex CLI 配置、认证、会话状态
├── miniforge3/             # 仅在 Docker 不可用时使用的 Conda 安装
├── conda-pkgs/             # Conda 包缓存
├── cache/                  # pip、Hugging Face、Torch、Triton 缓存
├── ray/                    # Ray 临时目录
├── datasets/
├── models/
├── checkpoints/
├── logs/
└── docker-images/          # Docker 镜像归档（.tar）
```

## 第一步：只做诊断

先设置持久目录变量并检查空间、GPU、Docker。以下命令不修改环境（`mkdir` 除外）：

```bash
export PERSIST=/data/L202500291
mkdir -p "$PERSIST"
df -h "$PERSIST"
id
nvidia-smi
command -v docker || true
docker version || true
docker info || true
```

如果 `docker info` 成功，再做 GPU 透传检查：

```bash
docker run --rm --gpus all nvidia/cuda:12.4.1-base-ubuntu22.04 nvidia-smi
```

若最后一条因网络拉取失败，记录错误即可；不要据此断定 Docker 或 GPU 不可用。
若镜像已存在但 `--gpus all` 失败，Docker 路线不可直接使用，需要切换到持久 Conda。

将以下信息（注意隐藏用户名、IP、token 等敏感内容）发给新开的 Codex 对话：

1. `df -h "$PERSIST"` 的输出；
2. `nvidia-smi` 的驱动与 CUDA 版本；
3. `docker version`、`docker info` 的成功或报错；
4. GPU 透传检查的成功或报错；
5. 是否能访问 GitHub、Hugging Face、NVIDIA NGC 或校内镜像源。

## 第二步：让 Codex 状态永久保存

无论是否使用 Docker，都需要在每个新 CCI 容器的 shell 中设置：

```bash
export CODEX_HOME=/data/L202500291/codex
export XDG_CACHE_HOME=/data/L202500291/cache/xdg
mkdir -p "$CODEX_HOME" "$XDG_CACHE_HOME"
chmod 700 "$CODEX_HOME"
```

首次迁移时，若 `/home/.codex` 已含有需要保留的登录状态，先复制而非删除：

```bash
cp -a "$HOME/.codex/." "$CODEX_HOME/"
```

然后把上述两个 `export` 写入 `/home/.zshrc`。这份 `.zshrc` 只需通过 CCI
“保存镜像”固化一次；之后 Codex 的实际可变状态都留在 `/data`，不用再保存镜像。
不要在未确认 `$CODEX_HOME` 已生效前删除 `/home/.codex`。

## 第三步：Docker 成功时的路线（首选）

Docker 成功的标准是：`docker info` 成功，并且 GPU 透传检查能看到 4 张 H100。

1. 把 Git 工作副本放在 `/data/L202500291/SimpleTIR`，不要只留在 `/home`。
2. 请新 Codex 对话在仓库内创建 **SimpleTIR 专用 Dockerfile 和启动脚本**。不要直接使用
   `docker/Dockerfile.ngc.vllm0.8`：它是通用 veRL 配方，安装 `vllm==0.8.2` 和最新
   PyPI `verl`，可能与本仓库的 vendored `verl` 不匹配。
3. 镜像构建成功后，将其归档到持久目录一次：

```bash
docker save -o "$PERSIST/docker-images/simpletir-cci.tar" simpletir:cci
```

4. 每次新 CCI 容器启动时，若 Docker 本地镜像丢失，自动恢复即可：

```bash
docker image inspect simpletir:cci >/dev/null 2>&1 || \
  docker load -i "$PERSIST/docker-images/simpletir-cci.tar"
```

Docker 运行时必须挂载 `/data/L202500291` 中的工作副本、缓存、模型、数据和输出目录。
这样环境依赖由镜像固定，所有工作结果由 `/data` 永久保存。Docker daemon 是否可把
`data-root` 改到 `/data` 取决于 CCI 权限；即使不能改，使用上述镜像归档也可恢复。

## 第四步：Docker 不可用时的路线（备用）

不要试图迁移旧的 `/home` Conda。新建一个完全位于 `/data` 的 Miniforge：

```text
/data/L202500291/miniforge3
/data/L202500291/miniforge3/envs/simpletir
/data/L202500291/conda-pkgs
```

在 `.zshrc` 中固定 `CONDA_PKGS_DIRS=/data/L202500291/conda-pkgs`，并从
`/data/L202500291/miniforge3/etc/profile.d/conda.sh` 激活 Conda。之后再由 Codex 按
SimpleTIR 与 H100 的版本组合创建环境；不要直接用仓库的裸 `requirements.txt` 盲装。

## 训练环境完成的验收顺序

1. Python 可以导入 `torch`、`vllm`、`ray`、`verl` 与 `recipe.simpletir`；
2. `torch.cuda.device_count()` 返回 4；
3. 单卡/小 batch 的 vLLM 生成能运行；
4. 本地 sandbox 能执行一条无害 Python 代码；
5. 才运行缩小版 SimpleTIR smoke training。

首个 smoke run 使用 7B、4 卡、`max_turns=3`、短上下文、`n=4`；不要一开始使用
README 的 `n=16`、长上下文和大 batch。
