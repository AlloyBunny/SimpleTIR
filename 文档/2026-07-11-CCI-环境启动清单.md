## 第一步：只做诊断

先设置持久目录变量并检查空间、GPU、Docker。以下命令不修改环境（`mkdir` 除外）：

```bash
export PERSIST=/data/L202500291
mkdir -p "$PERSIST"
df -h "$PERSIST"
id
nvidia-smi
command -v docker || true
docker version || true
docker info || true
```

如果 `docker info` 成功，再做 GPU 透传检查：

```bash
docker run --rm --gpus all nvidia/cuda:12.4.1-base-ubuntu22.04 nvidia-smi
```

若最后一条因网络拉取失败，记录错误即可；不要据此断定 Docker 或 GPU 不可用。
若镜像已存在但 `--gpus all` 失败，Docker 路线不可直接使用，需要切换到持久 Conda。

将以下信息（注意隐藏用户名、IP、token 等敏感内容）发给新开的 Codex 对话：

1. `df -h "$PERSIST"` 的输出；
2. `nvidia-smi` 的驱动与 CUDA 版本；
3. `docker version`、`docker info` 的成功或报错；
4. GPU 透传检查的成功或报错；
5. 是否能访问 GitHub、Hugging Face、NVIDIA NGC 或校内镜像源。

## 第二步：让 Codex 状态永久保存

文件系统迁移已完成，但本容器中仍有一个迁移前启动的服务待重启。新 CCI 容器启动后，先执行：

```bash

readlink -f "$HOME/.codex"
```

预期输出为 `/data/L202500291/.codex`。脚本会在持久目录不存在时迁移原状态；若持久目录
已存在，则保留意外出现的非链接状态为带时间戳的备份，避免覆盖。

## 第三步：Docker 成功时的路线（首选）

Docker 成功的标准是：`docker info` 成功，并且 GPU 透传检查能看到 4 张 H100。

1. 把 Git 工作副本放在 `/data/L202500291/SimpleTIR`，不要只留在 `/home`。
2. 请新 Codex 对话在仓库内创建 **SimpleTIR 专用 Dockerfile 和启动脚本**。不要直接使用
   `docker/Dockerfile.ngc.vllm0.8`：它是通用 veRL 配方，安装 `vllm==0.8.2` 和最新
   PyPI `verl`，可能与本仓库的 vendored `verl` 不匹配。
3. 镜像构建成功后，将其归档到持久目录一次：

```bash
docker save -o "$PERSIST/docker-images/simpletir-cci.tar" simpletir:cci
```

4. 每次新 CCI 容器启动时，若 Docker 本地镜像丢失，自动恢复即可：

```bash
docker image inspect simpletir:cci >/dev/null 2>&1 || \
  docker load -i "$PERSIST/docker-images/simpletir-cci.tar"
```

Docker 运行时必须挂载 `/data/L202500291` 中的工作副本、缓存、模型、数据和输出目录。
这样环境依赖由镜像固定，所有工作结果由 `/data` 永久保存。Docker daemon 是否可把
`data-root` 改到 `/data` 取决于 CCI 权限；即使不能改，使用上述镜像归档也可恢复。

## 第六步：本地 Firejail sandbox（推荐的单机验证方案）

SimpleTIR 的代码执行奖励需要设置 `SANDBOX_ENDPOINT`。本仓库的本地实现使用
FastAPI/Uvicorn 调用 Firejail；它仅支持 Python，并与
`http://127.0.0.1:12345/faas/sandbox/` 兼容。

### 当前 CCI 容器的限制（已验证）

当前 CCI 是受限的 containerd 容器，缺少 Firejail 建立嵌套 mount namespace 所需的
`CAP_SYS_ADMIN`；`unshare --mount` 返回 `Operation not permitted`。Firejail 还会检测到
已有容器 sandbox 并在某些调用中降级为不隔离执行。实测该降级路径中的代码能看到
`/data/L202500291/SimpleTIR/README.md`，因此不能作为安全的模型代码执行环境。

本地启动器会先运行真实隔离自检：在执行 `print(1 + 1)` 前断言 `/data` 不可见。当前 CCI
中该自检会失败，启动器以退出码 `1` 停止 Uvicorn 服务并保留失败记录，避免错误导出一个
未隔离的 `SANDBOX_ENDPOINT`。这是正确的安全行为；不要通过删除 profile、`--net=none` 或
隔离检查来绕过它。

在此 CCI 中完成**工具增强的完整复现**需要以下任一条件：

1. 提供兼容 Seed-Sandbox API 的远程 `SANDBOX_ENDPOINT`（推荐）；或
2. 使用允许 Firejail mount namespace 的特权/兼容容器。

`scripts/start_local_sandbox.sh` 是唯一的启动入口。它会：

1. 确认以 root 身份运行，安装仓库内持久化保存的 Firejail profile 到
   `/etc/firejail/sandbox.profile`；
2. 创建不可登录的 `sandboxer` 系统用户；
3. 将服务日志、PID 和运行时 HOME 写入 `/data/L202500291/sandbox/`；
4. 以 `sandboxer` 身份启动 4 个 Uvicorn worker，并且只监听回环地址；
5. 输出可直接 export 的 `SANDBOX_ENDPOINT`。

在满足上述内核权限条件的容器中，第一次启动前由管理员安装 Firejail；之后每次容器重启都执行：

```bash
source /data/L202500291/SimpleTIR/scripts/activate_simpletir_env.sh
sudo /data/L202500291/SimpleTIR/scripts/start_local_sandbox.sh
export SANDBOX_ENDPOINT=http://127.0.0.1:12345/faas/sandbox/
```

停止服务：

```bash
sudo /data/L202500291/SimpleTIR/scripts/stop_local_sandbox.sh
```

本地方案适合单机 smoke evaluation；它会与训练争抢 CPU，且项目只明确测试过单节点评估。
大规模 rollout 训练应改用独立的远程 sandbox 服务，并仅将其 URL 设置为
`SANDBOX_ENDPOINT`。当前 CCI 请直接使用远程服务，且 URL 必须以 `/faas/sandbox/` 结尾。

## 第七步：最小复现顺序

先验证 sandbox，不要直接启动 README 中的 8×H100 长上下文训练：

```bash
source /data/L202500291/SimpleTIR/scripts/activate_simpletir_env.sh
export SANDBOX_ENDPOINT=http://127.0.0.1:12345/faas/sandbox/
curl -fsS -X POST "$SANDBOX_ENDPOINT" \
  -H 'Content-Type: application/json' \
  -d '{"code":"print(1 + 1)","language":"python","compile_timeout":1,"run_timeout":3}'
```

成功后，先运行一条 4×H100、单样本、单回合的评估 smoke run：

```bash
source /data/L202500291/SimpleTIR/scripts/activate_simpletir_env.sh
export MODEL_PATH=/data/L202500291/code-r1/models
export DATA_PATH=/data/L202500291/SimpleTIR/datasets
export CHECKPOINT_PATH=/data/L202500291/outputs/simpletir/checkpoints
export LOG_PATH=/data/L202500291/outputs/simpletir/logs
export SANDBOX_ENDPOINT=http://127.0.0.1:12345/faas/sandbox/
export WANDB_MODE=offline
export NNODES=1 GPUS_PER_NODE=4 RESUME=False CONFIG_NAME=simpletir_trainer

bash train.sh \
  --model_name Qwen2.5-7B-Instruct-1M \
  --valid_dataset "deepscaler/aime" \
  --val_only True \
  --max_turns 1 \
  --max_prompt_length 4096 \
  --max_response_length 1024 \
  --val_sample_size 1 \
  --n_val 1 \
  --rollout_tp 2 \
  --sp_size 1
```

确认该评估完成后，再逐步提高验证样本数、回合数和输出长度；训练前应先进行小 batch
smoke run，不能直接采用 README 的 8 卡、`n=16` 与长上下文参数。

### 无工具基础链路 smoke（当前 CCI 可执行）

在远程 sandbox 尚未提供前，可以验证模型、Ray、vLLM、FSDP 和数据加载链路，但这**不等同于**
SimpleTIR 的工具增强复现。使用与上节相同的命令，并追加：

```bash
--tool_use False
```

该模式不会把模型生成的 Python 发送给 sandbox；它的唯一用途是确认 4×H100 上的基础评估
管线能够加载现有 Qwen2.5 7B 权重并完成一个验证样本。

2026-07-11 已完成一次该 smoke：退出码为 `0`，日志位于
`/data/L202500291/outputs/simpletir/logs/smoke_no_tool_score_timeout_fix_maxturn1_maxpro4096_maxres1024.log`。
该单个采样答案得分为 `0.0`，但 `is_filter=0.0`，说明它是正常评分出的错误答案，而非评分超时
生成的默认分数。

评分任务内部的数学验证仍限制为 5 秒，但首次启动 Ray worker 的调度和导入可能超过 5 秒。
因此 `MathRewardManager` 的外层结果等待已设置为 30 秒，避免把成功完成的冷启动任务误记为
默认零分；不要把该 30 秒值用于放宽数学验证本身的执行时间。

## 训练环境完成的验收顺序

1. Python 可以导入 `torch`、`vllm`、`ray`、`verl` 与 `recipe.simpletir`；
2. `torch.cuda.device_count()` 返回 4；
3. 单卡/小 batch 的 vLLM 生成能运行；
4. 本地 sandbox 能执行一条无害 Python 代码；
5. 才运行缩小版 SimpleTIR smoke training。

首个 smoke run 采用第七步的单样本评估命令；不要一开始使用 README 的 8 卡、`n=16`、
长上下文和大 batch。
